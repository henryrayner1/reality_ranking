# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Frontend (repo root):
- `npm run dev` — Vite dev server on port 5173, proxies `/api` to `http://localhost:4000` (see `vite.config.ts`)
- `npm run build` — `tsc -b && vite build`, then copies `dist/index.html` to `dist/404.html` (GitHub Pages SPA-fallback trick; production base path is `/reality_ranking/`, dev stays at `/`)
- `npm run lint` — ESLint

Backend (`backend/`):
- `npm run dev` — `tsx watch src/server.ts`; port comes from `.env` (`PORT`, defaults to 4000)
- `npm run build` / `npm start` — compile then run compiled JS
- `npm run prisma:generate` / `npm run prisma:migrate` — regenerate client / apply schema changes
- `npm run seed` — `tsx prisma/seed.ts`

Run both together from the root: `npm start` (runs `dev:frontend` and `dev:backend` in parallel via `npm-run-all`).

There is no test suite in either package.

`backend/.env` (`DATABASE_URL`/`DIRECT_DATABASE_URL`) points at a real, already-populated Postgres instance in normal development — check it before assuming you need the `docker-compose.yml` Postgres container, which is not what's actually wired up by default.

## Architecture

- Two-package layout: Vite/React/TypeScript frontend at the repo root, Express/Prisma backend in `backend/`. No shared package — types are duplicated by hand between `src/utils/Constants.ts` and the backend's Prisma schema/route response shapes.
- Data model (`backend/prisma/schema.prisma`): `Show` → `Season` → (`Contestant`, `Episode`); `Episode` → `Elimination` / `Ranking`. A `Contestant` can have at most one `Elimination` (`Elimination.contestantId` is `@unique`).
- No auth middleware or token: `POST /api/users/login` just returns the user row, and the client keeps it in the Redux `user` slice (persisted to localStorage manually and deliberately kept *out* of redux-persist — see the comment in `src/redux/store.ts` explaining the two-competing-copies bug that caused). Admin-only UI is gated purely by a client-side `accountType` check.
- Redux (`src/redux/`) now holds only the `user` slice — `shows`/`seasons`/`episodes`/rankings all moved to React Query (`src/hooks/queries.ts`) as the single source of truth for server data, since the old Redux-vs-React-Query split let Admin edits silently go stale on the Ranking/Insights pages until a hard reload. `GET /api/shows/` returns the *entire* nested tree in one call (every show → its seasons → each season's `contestants[]` and `episodes[]`, each episode's `eliminations[]`) — every hook in `queries.ts` (`useShows`, `useSeasons`, `useEpisodesByShow`, `useShowTree`) derives from that single `['shows']` query rather than hitting separate endpoints, and every mutation across `Admin*.tsx` invalidates just `['shows']`. **Gotcha:** `GET /api/shows/:showId` (used nowhere anymore, but easy to reach for) returns a much sparser `Show` — its `seasons[]` omit `contestants` entirely — so a hook built against that endpoint will silently produce `season.contestants === undefined` instead of an empty array, crashing anything that does `.contestants.length`. Don't add a new hook against `/api/shows/:showId` without checking its response shape first. `QueryClient` (`src/main.tsx`) is configured with `staleTime: Infinity`/`gcTime: Infinity`/no background refetch to mirror Redux's old "fetch once, update only via explicit action" model — data only changes via a mutation's `invalidateQueries`.
- A page that derives "current season" from a `useState` selected-id plus a `useEffect` that resets it when the show changes (e.g. `Insights.tsx`'s old `selectedSeasonId`) will render one frame with the *previous* show's season id after the show itself has already updated — `useEffect` only runs after commit/paint, so that stale id briefly finds nothing in the new show's `seasons[]`. If a child component treats "no current season" as its own independent loading signal (e.g. `InsightsRankingTable`/`ContestantDetailPanel`'s `if (loadingFlag || !currSeason)` check), that frame shows a loading-circle flash even though every query involved is already fully cached (`staleTime: Infinity`) and nothing is actually loading. Fix by deriving the fallback season directly in the render body (same "current season, else last season" logic the effect uses) instead of only setting it via the effect — the effect can still exist to persist the choice into state for controls like `SeasonSelect`, but the *derived* value used for rendering should never depend on that state having caught up yet.
- The "currently selected show" is a URL param, not client state: `/ranking/:showSlug`, `/insights/:showSlug`, `/admin/:showSlug`, where the slug is `slugifyShowName(show.name)` (`src/utils/slug.ts`) — not the show's id, so URLs read as `/ranking/rupauls-drag-race` rather than a cuid. There's no persisted `slug` column; it's derived from `name` on every read, so **two shows with the same name would collide** (fine at this app's scale, but worth knowing if that ever changes — would need a real stored/unique slug field). Every page resolves `showSlug` back to a real `Show.id` via `shows.find(s => slugifyShowName(s.name) === showSlug)` before handing that id to the data hooks (`useShowTree(showId)` etc. still take the real id, unchanged) — don't skip that resolution step when adding a new show-scoped route or `navigate()` call, or you'll end up passing a slug where an id is expected. A bare `/ranking`, `/insights`, or `/admin` (no slug) renders `DefaultShowRedirect`, which waits for `useShows()` and redirects to the first loaded show's slug. `ShowSelect` itself is still id-based internally (`shows`, `currShowId`, `onSelectShow(showId)`) — each page's `onSelectShow` looks the clicked id back up in `shows` and slugifies before calling `navigate()`.
- Two ranking modes per show (`Show.rankingMode`: `EPISODE` | `DAILY`). `EPISODE` shows open ranking a fixed duration after `airDate`. `DAILY` shows (e.g. Big Brother) open ranking per America/New_York calendar day instead of per-episode; that timezone/day-boundary math lives in `src/utils/episodeRankability.ts` and is **duplicated by hand** in `backend/src/utils/episodeRankability.ts` and `backend/src/utils/dailyEpisode.ts` (which auto-creates each day's Episode row) — keep both sides in sync, per the comments in each file.
- `GET /api/eliminations/` (`getEliminations()`) returns every elimination across *every* show/season the logged-in user has, grouped only by `episodeId` — it is not scoped to a season. Anything deriving "who's eliminated as of episode N" must filter to the current season first (see `currentSeasonEliminations` in `RankingComponent2.tsx`), otherwise episode-number collisions across shows (nearly every show has an "episode 1") leak the wrong show's contestants into a grid. **This is a different shape from `GET /api/eliminations/bySeason/:seasonId`** (`useEliminationsBySeason`): the unscoped endpoint groups by episode (`{episodeId, episodeNumber, seasonId, contestantIds: string[]}`), while the season-scoped one returns one row per elimination (`EliminationEntry`: `{id, episodeId, contestantId, eliminationType, episode}`). `ContestantDetailPanel` (and anything else built around per-contestant `eliminationType`) needs the latter — don't feed it the grouped shape.
- `RankingComponent2.tsx`'s "Rankings Table" / "By Contestant" tabs mirror `Insights.tsx`'s, but "By Contestant" reuses the *same* `ContestantDetailPanel`/`ContestantTrendChart` components fed synthetic `InsightsResponse` objects built from only the logged-in user's own `favoriteRankings`/`winnerRankings` (`buildOwnInsights` in `RankingComponent2.tsx`), not `useRankingsInsights` — the ranking page has no reason to show the season-wide average since it's not admin/analytics-facing, just the viewer's own submitted line. `averagePosition` there is just that single ranking's 1-indexed position (`contestantIds` index + 1), not an actual average.
- Eliminations are immutable at the bottom of the ranking column starting the *same* episode/day a contestant is eliminated (deliberate product decision, not a bug) — both `Episode.tsx` (live ranking) and `buildPastRankingColumn` (`src/utils/util.ts`, past rankings) compute one unconditional `getEliminationOrder(eliminations, episodeNumber)` threshold and use it both to exclude the contestant from the draggable/active list and to place them in the eliminated bucket. An earlier "lock in place" variant (contestant stays at their submitted rank position through their own elimination episode, only moving to the bottom starting the next one) was tried and explicitly reverted — don't reintroduce that split without being asked.
- `buildPastRankingColumn` (`src/utils/util.ts`) fills in any roster member missing from a past `Ranking.contestantIds` (e.g. historical data gaps) as an active fallback entry — but a contestant added to the season *after* a given episode already happened must not be backfilled into that episode's column, only into later ones. This is decided by comparing `Contestant.createdAt` to the episode's `airDate` (or, for DAILY shows, calendar day via `getTodayDayKey`) — see `wasContestantOnRosterFor` in `RankingComponent2.tsx`. The function takes two separate id lists for this reason: `seasonContestantIds` (full current roster, used only to pad every column to the same row count) and `eligibleContestantIds` (roster as of that episode, used for the active-fallback step) — don't collapse them back into one list.
- The ranking grid (`RankingComponent2.tsx` + `Episode.tsx`) and the insights table (`InsightsRankingTable.tsx`) both lay out contestant-by-episode data as one CSS Grid with `grid-auto-flow: column` and fixed-width columns — not a `<table>`. Two rules keep this from silently misaligning column-by-column:
  1. Every column must contribute the exact same number of grid cells; pad short columns with blank/null placeholder cells instead of returning a variable-length array (see `buildPastRankingColumn` in `src/utils/util.ts`).
  2. A column's cells must be direct children of the grid, never wrapped in a container div with an explicit `gridRow: span(...)` — a spanning wrapper stops a cell's real content height from growing its row's `1fr` track, so cells silently render smaller than the rank column.
- Drag-and-drop ranking (`Episode.tsx`) uses `@dnd-kit/core` + `@dnd-kit/sortable`. `DndContext`/`SortableContext` render no DOM element of their own (pure context providers), so they can wrap grid cells without violating the "direct children" rule above.
- Contestant photos fall back to a naming convention when no `photoUrl` is set: `getImagePath` in `ContestantIcon.tsx` builds `/uploads/<show-slug>/season_<n>/contestants/<contestant-name>.png` and passes it through `backendUrl()` — same as the explicit-`photoUrl` case, both resolve to a backend-served URL, not a frontend asset import. Uploaded images are written to disk under `backend/uploads` (`uploadsDirectory = path.resolve(process.cwd(), "uploads")` in `backend/src/routes/images.ts` and `show.ts`, relying on `process.cwd()` being `backend/` in both dev and prod) and served statically at `/uploads` (`backend/src/server.ts`). The `/uploads` URL prefix is independent of where the directory physically lives — stored `photoUrl` values and the frontend's constructed paths only ever reference the URL, never the disk location, so the backing directory can be moved freely as long as `express.static()` and the two `uploadsDirectory` resolutions stay in sync and point at the same place.
